Phase 1 · Step 1 — Salesforce calls Genesys to retrieve call data
1
Scheduled Apex calls Genesys Analytics API and Recordings API
Named Credential: Genesys · Apex Callout
Genesys API

The Scheduled Apex job calls the Genesys Analytics API to fetch completed voice calls in the last time window, then calls the Recordings API per call to get the audio file URL.

// Phase1UploadJob.apex — Scheduled Apex entry point global class Phase1UploadJob implements Schedulable { global void execute(SchedulableContext sc) { Phase1Processor.run(); } } public class Phase1Processor { public static void run() { // Step 1a: call Genesys Analytics API String genesysToken = GenesysAuth.getToken(); String interval = buildInterval(); // last 1 hour ISO window String body = '{"interval":"' + interval + '","order":"asc","orderBy":"conversationStart",' + '"paging":{"pageSize":20,"pageNumber":1},' + '"segmentFilters":[{"type":"and","predicates":[{"type":"dimension",' + '"dimension":"mediaType","operator":"matches","value":"voice"}]}]}'; HttpRequest req = new HttpRequest(); req.setEndpoint('callout:Genesys_NC/api/v2/analytics/conversations/details/query'); req.setMethod('POST'); req.setHeader('Content-Type', 'application/json'); req.setHeader('Authorization', 'Bearer ' + genesysToken); req.setBody(body); HttpResponse res = new Http().send(req); Map<String,Object> parsed = (Map<String,Object>) JSON.deserializeUntyped(res.getBody()); List<Object> conversations = (List<Object>) parsed.get('conversations'); for (Object c : conversations) { Map<String,Object> conv = (Map<String,Object>) c; processConversation(conv, genesysToken); } } }
Named Credential: Genesys_NC must be configured in Salesforce Setup → Named Credentials with the Genesys OAuth 2.0 client credentials. The callout URL then becomes callout:Genesys_NC/api/v2/... — no hardcoded URLs or tokens in Apex.

Using the clientUniqueId from the Genesys call, Salesforce queries its own Task table directly. No REST API required — this is a native SOQL query inside the same Apex transaction.

// Step 2: find Task by clientUniqueId, follow chain to Account + Opportunity String clientUniqueId = '3b10d6d6-13be-abcd'; // from Genesys participant attributes List<Task> tasks = [ SELECT Id, WhoId, Who.AccountId FROM Task WHERE clientUniqueId__c = :clientUniqueId LIMIT 1 ]; if (tasks.isEmpty()) { // No matching Task — log and skip this call logError(clientUniqueId, 'No Task found for clientUniqueId'); return; } Task t = tasks[0]; Id contactId = t.WhoId; // e.g. 003XXX Id taskId = t.Id; // e.g. 00TAAA — stored for Phase 2 // Follow chain: Contact → Account → open Opportunity List<Contact> contacts = [ SELECT Id, AccountId, Account.Name, (SELECT Id, Name, StageName FROM Opportunities WHERE IsClosed = false ORDER BY LastModifiedDate DESC LIMIT 1) FROM Contact WHERE Id = :contactId LIMIT 1 ]; Contact con = contacts[0]; Id accountId = con.AccountId; Id opportunityId = con.Opportunities.isEmpty() ? null : con.Opportunities[0].Id; // These IDs go into the Gong context block in Step 4
Phase 1 · Step 3 — Resolve Genesys User ID to Gong Numeric ID
Email cannot be used in any API call. Gong's GET /v2/users API response includes emailAddress for every user — calling it to match by email would bring email into the Apex transaction, violating the customer's security policy. The only compliant approach is to populate Gong_Numeric_Id__c on Salesforce User records manually, through a one-time admin action in the Gong UI, before any calls are processed.
How to find a Gong user's numeric ID: A Gong Technical Admin navigates to Gong Admin Center → Users, opens any user's profile page, and copies the numeric ID from the browser URL. The URL will look like: https://app.gong.io/settings/users/7782342274025937895/profile. The long number in the URL path is the Gong Numeric ID. This value is then entered directly into the Gong_Numeric_Id__c field on the corresponding Salesforce User record. No API call is made — no email is involved.

Case A — Already mapped (runtime path)

Salesforce User has Gong_Numeric_Id__c populated. Apex reads it directly. Zero external API calls. This is the only path that runs automatically during call processing.

Case B — Gong ID missing (blocked)

If Gong_Numeric_Id__c is blank, Apex cannot resolve the ID automatically without calling the Gong API (which would expose email). The call is skipped and logged. An admin must populate the field manually before retrying.

Case C — No mapping at all (blocked)

Neither Genesys_User_Id__c nor Gong_Numeric_Id__c is set on the User. Both must be populated by an admin before this agent's calls can be processed. Apex logs the gap and skips the call.

Pre-requisite before go-live: A Gong admin and a Salesforce admin must complete a one-time mapping exercise — match every Genesys agent to their Salesforce User record, retrieve each agent's Gong Numeric ID from the Gong UI URL, and enter both values onto the User record. Only then will Apex be able to process calls for those agents.

Step-by-step for each agent:

  1. 1 Gong Admin logs into Gong → Admin Center → Users → opens the agent's profile page
  2. 2 Copies the numeric ID from the browser URL — e.g. https://app.gong.io/settings/users/7782342274025937895/profile

// Step 3: resolve Genesys userId → Gong Numeric Id // Reads directly from the Salesforce User record. // No API calls to Gong or Genesys. No email used at any point. // Gong_Numeric_Id__c must be pre-populated by an admin (see admin task above). public static String resolveGongUserId(String genesysUserId) { List<User> users = [ SELECT Id, Gong_Numeric_Id__c, Gong_Mapping_Status__c FROM User WHERE Genesys_User_Id__c = :genesysUserId LIMIT 1 ]; if (users.isEmpty()) { // No Salesforce User has this Genesys User ID — admin must add it logError(genesysUserId, 'No SF User found for Genesys ID. Admin must set Genesys_User_Id__c.'); return null; } User sfUser = users[0]; if (String.isBlank(sfUser.Gong_Numeric_Id__c)) { // SF User found but Gong ID not yet populated — admin must add it from Gong UI logError(genesysUserId, 'Gong_Numeric_Id__c is blank for User ' + sfUser.Id + '. Admin must retrieve the Gong Numeric ID from the Gong UI and enter it on this User record.' ); return null; } // Gong Numeric ID is present — return it directly, no API call needed return sfUser.Gong_Numeric_Id__c; }
Once mapped, always fast: Every call for a pre-mapped agent hits a single SOQL query and returns in milliseconds. No external callouts, no email, no Gong API call. If an agent's Gong account changes, the admin updates Gong_Numeric_Id__c on their Salesforce User record directly.

// Step 5: POST to Gong and store the returned callId on the Task record HttpRequest req = new HttpRequest(); req.setEndpoint('callout:Gong_NC/v2/calls'); req.setMethod('POST'); req.setHeader('Content-Type', 'application/json'); req.setBody(JSON.serialize(payload)); HttpResponse res = new Http().send(req); if (res.getStatusCode() == 200) { Map<String,Object> result = (Map<String,Object>) JSON.deserializeUntyped(res.getBody()); String gongCallId = (String) result.get('callId'); // Store GongCallId__c on the Task — this is what Phase 2 webhook uses Task t = new Task(Id = taskId, GongCallId__c = gongCallId); update t; } else { // Log error — do not throw, allow other calls in batch to continue logError(conversationId, 'Gong upload failed: ' + res.getBody()); }
Why store GongCallId__c on the Task: When Gong fires the Phase 2 webhook, it sends the metaData.id (Gong call ID) and the clientUniqueId. Both values are stored on the Task — Salesforce can match the incoming webhook to the right Task using either. The clientUniqueId is the primary key since it was set at call creation time.

Phase 2 · Salesforce polls Gong and writes analysis back to Task

A Scheduled Apex job runs 2–4 hours after Phase 1. Salesforce calls the Gong API to retrieve completed call analysis, matches each call back to its Salesforce Task via clientUniqueId, and updates the Task natively using Apex DML. No external REST PATCH to Salesforce is needed — Salesforce is already running the code.

Why Scheduled Apex instead of Gong webhooks: Gong's webhook payload includes parties[].emailAddress for all call participants. Since email cannot appear in any API payload under the customer's security policy, the webhook approach is not suitable. Salesforce initiates the data pull on a schedule instead — no inbound payload containing email is ever received.

Phase 2 · Step 1 — Poll Gong for calls with completed analysis
1
Scheduled Apex calls GET /v2/calls to retrieve processed calls
Named Credential: Gong_NC · runs 2–4 hrs after Phase 1
Gong API

The job queries Gong for calls in the same time window as Phase 1. It filters for calls where Gong has completed processing and skips any Task already marked as synced.

// Phase2AnalysisJob.apex — Scheduled Apex entry point global class Phase2AnalysisJob implements Schedulable { global void execute(SchedulableContext sc) { Phase2Processor.run(); } } public class Phase2Processor { public static void run() { // Time window: calls from 2–4 hours ago (allowing Gong processing time) Datetime toTime = Datetime.now().addHours(-2); Datetime fromTime = Datetime.now().addHours(-4); String fromISO = fromTime.formatGmt('yyyy-MM-dd'T'HH:mm:ss'Z''); String toISO = toTime.formatGmt('yyyy-MM-dd'T'HH:mm:ss'Z''); HttpRequest req = new HttpRequest(); req.setEndpoint('callout:Gong_NC/v2/calls?fromDateTime=' + fromISO + '&toDateTime=' + toISO); req.setMethod('GET'); HttpResponse res = new Http().send(req); Map<String,Object> data = (Map<String,Object>) JSON.deserializeUntyped(res.getBody()); List<Object> calls = (List<Object>) data.get('calls'); // Collect call IDs for batch analysis fetch in Step 2 List<String> callIds = new List<String>(); Map<String,String> callIdToClientUniqueId = new Map<String,String>(); Map<String,String> callIdToUrl = new Map<String,String>(); for (Object c : calls) { Map<String,Object> call = (Map<String,Object>) c; String gongCallId = (String) call.get('id'); String clientUniqueId = (String) call.get('metaData.clientUniqueId'); String callUrl = (String) call.get('url'); // Skip calls already synced — avoids re-processing List<Task> existing = [ SELECT GongAnalysisSyncStatus__c FROM Task WHERE clientUniqueId__c = :clientUniqueId LIMIT 1 ]; if (!existing.isEmpty() && 'Synced'.equals(existing[0].GongAnalysisSyncStatus__c)) continue; callIds.add(gongCallId); callIdToClientUniqueId.put(gongCallId, clientUniqueId); callIdToUrl.put(gongCallId, callUrl); } if (!callIds.isEmpty()) { fetchAnalysisAndUpdate(callIds, callIdToClientUniqueId, callIdToUrl); } } }
POC volume cap: Limit to 5–20 calls per run for the POC. The GET /v2/calls response is paginated — use the cursor field for full production pagination. For the POC a single page is sufficient.
Phase 2 · Step 2 — Fetch detailed analysis per call
2
POST /v2/calls/extensive — retrieve AI summary, topics, talk ratio
Named Credential: Gong_NC · batched by call ID array · parties explicitly excluded
Gong API · POST

A single POST call fetches structured analysis for all call IDs collected in Step 1. The contentSelector.exposedFields controls exactly which sections Gong returns in the response. parties must be omitted from exposedFields — including it would cause Gong to return emailAddress for every participant, including internal Gong users and external call participants.

Why POST, not GET: GET /v2/calls (Step 1) is a GET request — date filters go in the URL as query parameters. POST /v2/calls/extensive (this step) is a POST request that requires a JSON body containing the call ID filter and content selector. The two endpoints have different HTTP methods by design.
Excluding email from the response: When parties: true is included in exposedFields, Gong returns an array of all call participants including their emailAddress. To prevent any email from appearing in this API response, parties is deliberately not included in the request. Gong will then return only the analysis sections requested — content and interaction — with no party contact data at all.
// Step 2: POST /v2/calls/extensive — fetch analysis for collected call IDs // parties is deliberately omitted from exposedFields — this prevents Gong // from returning emailAddress for any participant (internal or external) public static void fetchAnalysisAndUpdate( List<String> callIds, Map<String,String> callIdToClientUniqueId, Map<String,String> callIdToUrl) { // exposedFields controls what Gong includes in the response. // 'parties' is intentionally NOT listed here — omitting it means Gong // returns no party data and therefore no emailAddress fields at all. Map<String,Object> requestBody = new Map<String,Object> { 'filter' => new Map<String,Object>{ 'callIds' => callIds }, 'contentSelector' => new Map<String,Object>{ 'exposedFields' => new Map<String,Object>{ 'content': new Map<String,Object>{ 'topics': true, 'brief': true }, 'interaction': new Map<String,Object>{ 'speakers': true } // 'parties': true <-- deliberately excluded: would return emailAddress } } }; HttpRequest req = new HttpRequest(); req.setEndpoint('callout:Gong_NC/v2/calls/extensive'); req.setMethod('POST'); // /v2/calls/extensive is a POST endpoint req.setHeader('Content-Type', 'application/json'); req.setBody(JSON.serialize(requestBody)); HttpResponse res = new Http().send(req); Map<String,Object> data = (Map<String,Object>) JSON.deserializeUntyped(res.getBody()); List<Object> calls = (List<Object>) data.get('calls'); for (Object callObj : calls) { Map<String,Object> call = (Map<String,Object>) callObj; Map<String,Object> meta = (Map<String,Object>) call.get('metaData'); String gongCallId = (String) meta.get('id'); String clientUniqueId = callIdToClientUniqueId.get(gongCallId); String callUrl = callIdToUrl.get(gongCallId); Map<String,Object> content = (Map<String,Object>) call.get('content'); Map<String,Object> interaction = (Map<String,Object>) call.get('interaction'); String summary = (String) content?.get('brief'); String topics = joinTopics((List<Object>) content?.get('topics')); Decimal talkRatio = calcTalkRatio((List<Object>) interaction?.get('speakers')); updateTask(gongCallId, callUrl, clientUniqueId, summary, topics, talkRatio); } } // Helper: join topic names into comma-separated string private static String joinTopics(List<Object> topicsList) { if (topicsList == null) return ''; List<String> names = new List<String>(); for (Object t : topicsList) { Map<String,Object> topic = (Map<String,Object>) t; names.add((String) topic.get('name')); } return String.join(names, ', '); } // Helper: calculate agent talk % of total // speakers[] here contains only userId + talkTime — no emailAddress, // because parties was excluded from the request private static Decimal calcTalkRatio(List<Object> speakers) { if (speakers == null) return null; Decimal agentTime = 0, totalTime = 0; for (Object s : speakers) { Map<String,Object> spk = (Map<String,Object>) s; Decimal t = (Decimal) spk.get('talkTime'); totalTime += t; if (spk.get('userId') != null) agentTime += t; } return totalTime > 0 ? (agentTime / totalTime * 100).setScale(0) : null; }
Phase 2 · Step 3 — Match Salesforce Task via clientUniqueId
3
SOQL query on Task object — no external store needed
Salesforce is the database — direct query, no API
SOQL

The clientUniqueId returned by Gong's /v2/calls/extensive response is the same value stored on the Salesforce Task as clientUniqueId__c — set during Phase 1. Salesforce queries its own database directly.

// Step 3: match Task by clientUniqueId — called from Step 2 loop private static void updateTask( String gongCallId, String callUrl, String clientUniqueId, String summary, String topics, Decimal talkRatio) { if (clientUniqueId == null) { logError(gongCallId, 'clientUniqueId missing in Gong response'); return; } List<Task> tasks = [ SELECT Id, GongAnalysisSyncStatus__c FROM Task WHERE clientUniqueId__c = :clientUniqueId LIMIT 1 ]; if (tasks.isEmpty()) { logError(gongCallId, 'No Task found for clientUniqueId: ' + clientUniqueId); return; } // Skip if already synced — prevents duplicate processing on re-run if ('Synced'.equals(tasks[0].GongAnalysisSyncStatus__c)) { return; } // Proceed to Step 4 — DML update Task t = tasks[0]; t.GongCallId__c = gongCallId; t.GongCallUrl__c = callUrl; t.GongCallSummary__c = summary; t.GongTopics__c = topics; t.GongTalkRatio__c = talkRatio != null ? String.valueOf(talkRatio) : null; t.GongAnalysisSyncStatus__c = 'Synced'; t.GongAnalysisSyncedAt__c = Datetime.now(); try { update t; } catch (DmlException e) { t.GongAnalysisSyncStatus__c = 'Error'; t.GongAnalysisSyncError__c = e.getMessage(); update t; } }
clientUniqueId as the join key: This is the same value set in Phase 1 — the Genesys conversationId stored as clientUniqueId__c on the Task and passed to Gong as clientUniqueId in the upload payload. Gong returns it in the /v2/calls/extensive response, closing the loop without any intermediate mapping store.

Apex DML

The DML update executes inside the same Apex transaction — no REST API call, no authentication, no PATCH endpoint. Salesforce writes to its own database directly. The update statement is already shown in Step 3's updateTask() method — it is the final update t; call.

Fields written to Task on each successful sync
GongCallId__cGong numeric call ID
GongCallUrl__cDirect link to call in Gong
GongCallSummary__cAI-generated summary
GongTopics__cComma-joined topic list
GongTalkRatio__cAgent talk % of call

GongAnalysisSyncedAt__cTimestamp of this run
Error handling: If the DML fails, the Task is updated with GongAnalysisSyncStatus__c = 'Error' and the error message written to GongAnalysisSyncError__c. On the next Scheduled Apex run, the record is retried — the sync status check filters out 'Synced' records only, so 'Error' records are picked up again automatically.

Governor limits — what to watch for in the POC

Both phases run inside Salesforce transactions. The key limits are callouts per transaction (100 max) and the requirement to keep callouts out of loops. Phase 2 webhook is one callout-free transaction.

LimitValuePhase 1 impactPhase 2 impactSafeguard
HTTP callouts per transaction 100 ~4 callouts per call (Genesys analytics, recordings, Gong users if needed, Gong upload) 2 callouts total: GET /v2/calls + POST /v2/calls/extensive — keep batch small Process 5–20 calls max per run. Do NOT call Gong inside a loop — batch all IDs into one /v2/calls/extensive request
Callout timeout total 120s 4 calls × 3s each = well within limit 2 callouts — well within limit Set individual callout timeout to 20s
SOQL queries per transaction 100 ~3 queries per call (Task, Contact, User) 1 query per call (Task by clientUniqueId) — avoid SOQL inside loop by collecting IDs first Collect all clientUniqueIds then query in one WHERE clientUniqueId__c IN :ids
DML statements 150 1–2 (Task update, User update if remapping) 1 per call — bulk with update taskList; Collect all Task updates then DML once with a list
@future callouts Not allowed N/A Phase 2 runs in a Scheduled Apex context — callouts are permitted No issue — Scheduled Apex supports callouts natively
POC volume cap: Process a maximum of 5–20 call records per Phase 1 Scheduled Apex run. Filter by a narrow time window and explicitly skip Tasks where GongCallId__c is already populated. For full production scale, Phase 1 needs a Queueable Apex chain pattern to handle bulk volumes.
